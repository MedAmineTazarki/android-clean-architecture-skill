Patterns

Briques standards

Chaque pattern a un rôle précis. Combinés ensemble, ils forment une architecture robuste et testable.

Domain Model

Pure Kotlin data class — zéro dépendance framework. Le cœur métier de l'application.

Repository Interface

Contrat défini dans le domain. Implémenté par le data layer. Le ViewModel ne voit que l'interface.

UseCase

operator fun invoke() — encapsule une opération métier. Optionnel, utilisé si logique partagée ≥2 VMs.

Mapper

Convertit DTO ↔ Domain. Deux styles : object Mapper ou extension functions. Jamais dans le ViewModel.

Repository Impl

Implémentation concrète avec runCatching. Coordonne remote + local DataSources.

DTO

@Serializable data class qui reflète le JSON API. Isolé dans data/remote/dto/.

ViewModel

State holder injectant des interfaces domain via Koin. Expose un StateFlow<UiState>.

UiState

sealed interface : Loading | Success | Error. Toujours exhaustif dans le when.

Screen

Composable stateless. Reçoit des données + callbacks. Séparé du Route (connecté au VM).

Koin Module

Câblage interface → impl au runtime. single pour les repos, viewModel pour les VMs.

Error Handling

Result<T> + runCatching ou custom AppResult<T> sealed type pour des erreurs typées.

Fake Repository

Test double qui implémente l'interface domain. Pas de mocking framework — des Fakes purs.

Exemples concrets

Du domain model au ViewModel — chaque layer a un style distinct.

domain/model/Match.kt Domain 
// Zéro import framework
data class Match(
    val homeTeam: Club,
    val awayTeam: Club,
    val homeScore: Int,
    val awayScore: Int,
    val status: MatchStatus,
)

enum class MatchStatus {
    NOT_STARTED, LIVE, FINISHED
}
data/remote/mapper/MatchMapper.kt Data 
object MatchMapper {
    fun toDomain(dto: FixtureDto): Match {
        return Match(
            homeTeam = mapTeam(dto.teams.home),
            awayTeam = mapTeam(dto.teams.away),
            homeScore = dto.goals.home ?: 0,
            awayScore = dto.goals.away ?: 0,
            status = mapStatus(dto.status),
        )
    }
}
ui/viewmodel/MatchesViewModel.kt UI 
class MatchesViewModel(
    private val repository: MatchRepository,
) : ViewModel() {

    val uiState = MutableStateFlow<UiState>(Loading)

    fun loadMatches(date: LocalDate) {
        viewModelScope.launch {
            repository.getMatches(date)
                .onSuccess { uiState.value = Success(it) }
                .onFailure { uiState.value = Error(it) }
        }
    }
}
di/AppModules.kt DI 
val repositoryModule = module {
    single<MatchRepository> {
        MatchRepositoryImpl(get())
    }
    single<NewsRepository> {
        NewsRepositoryImpl(get())
    }
}

val viewModelModule = module {
    viewModel { MatchesViewModel(get()) }
    viewModel { NewsViewModel(get()) }
}

Logique métier dans @ComposableExtraire dans ViewModel ou UseCase

Checklist

Nouvelle feature

Les étapes à suivre pour chaque nouvelle fonctionnalité.

  • Domain model créé (pure Kotlin)
  • Repository interface dans domain/
  • UseCase (optionnel — si logique partagée)
  • DTOs avec @Serializable
  • Mapper DTO → Domain
  • Repository impl avec runCatching
  • UiState sealed class
  • ViewModel injecte interface domain
  • Screen stateless @Composable
  • Koin bindings mis à jour
  • Loading + Success + Error
  • Error handling avec Result<T>
